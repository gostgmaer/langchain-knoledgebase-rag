## DB session sharing bug

`packages/infrastructure/container/database.py`'s `session` provider was `providers.Resource` ŌĆö a singleton for the app's entire lifetime, meaning every request shared the exact same `AsyncSession`. Swapped to `providers.Factory` (a new session per resolution). This surfaced two further real problems, both fixed in the same pass:

- **A connection leak.** Naively making `session` a `Factory` meant every repository and the memory store independently opened their *own* fresh, never-closed connection per request. Fixed by rewriting `packages/api/dependencies.py`'s `get_conversation_manager` (later generalized into `request_scoped_session`/`get_scoped_container`) to open exactly one session per request, temporarily `.override()` the whole DI tree onto it, and explicitly close it in a `finally`.
- **A knock-on `TypeError`** in `packages/api/lifespan.py` ŌĆö `await container.graph.builder()` stopped being valid once the dependency chain became fully synchronous. Fixed by dropping the `await`.

## The checkpointer msgpack warning ŌĆö fixed, not just noted

LangGraph's checkpointer had been logging a warning on every single turn:

```
Deserializing unregistered type packages.planner.models.Capability from checkpoint.
This will be blocked in a future version. Set LANGGRAPH_STRICT_MSGPACK=true to block
now, or add to allowed_msgpack_modules to allow explicitly.
```

LangGraph's checkpoint serializer (`JsonPlusSerializer`) uses `ormsgpack` to serialize graph state between steps, including this codebase's own custom types (`Capability`, `ExecutionStep`, `ExecutionPlan` from `packages/planner/models.py`) and a third-party one (`asyncpg.pgproto.pgproto.UUID`). By default it allows any type through with just a warning ŌĆö but the warning itself says that's changing, and setting `LANGGRAPH_STRICT_MSGPACK=true` (or a future LangGraph default change) would turn this into a hard deserialization failure.

**Fixed** in `packages/graph/builder.py` ŌĆö instead of `MemorySaver()` with its default serializer, it now constructs one explicitly:

```python
from langgraph.checkpoint.serde.jsonplus import JsonPlusSerializer

_CHECKPOINT_SERDE = JsonPlusSerializer().with_msgpack_allowlist([
    Capability,
    ExecutionStep,
    ExecutionPlan,
    ("asyncpg.pgproto.pgproto", "UUID"),
])

self._checkpointer = MemorySaver(serde=_CHECKPOINT_SERDE)
```

`with_msgpack_allowlist` accepts either an actual class (it derives the `(module, name)` key automatically) or an explicit `(module, name)` tuple ŌĆö used the tuple form for the third-party asyncpg type since importing its internal `pgproto` module directly felt more fragile than just naming it.

**Verified live:** two full chat turns, zero occurrences of the warning that previously fired on every single one.

## The checkpointer was empty every request, and the earlier msgpack fix was a no-op the whole time

Asked to check on the LangGraph phase's partial items, tracing Checkpointing found a real bug ŌĆö and revealed an earlier fix this session had never actually done anything.

`GraphManager._config()` (`packages/graph/manager.py`) has always correctly passed `configurable.thread_id = str(state["thread_id"])` on every call ŌĆö that part was fine, contradicting an old doc claim that thread IDs "weren't wired." The actual problem: `GraphBuilder.__init__` (`packages/graph/builder.py`) constructed a fresh `MemorySaver()` every time it ran, and both `builder` and `manager` are DI-wired as `providers.Factory` ŌĆö so every single HTTP request got a brand-new, empty checkpoint store. LangGraph-level checkpointing provided zero benefit across turns; conversation continuity worked anyway, but only via `ConversationContextBuilder` reloading message history from the database, never via the checkpointer itself.

Fixed by pulling the checkpointer construction out of `GraphBuilder` entirely and wiring it as its own `providers.Singleton` in `packages/infrastructure/container/graph.py`, injected into `GraphBuilder` as a constructor parameter. This is safe in a way the earlier `Singleton`-caused memory bug wasn't: `MemorySaver` holds no per-request DB session, just an in-memory dict keyed by thread ID ŌĆö nothing here can go stale the way a cached DB session did.

**Fixing this then exposed that this session's earlier msgpack-allowlist fix had been inert from the start.** `JsonPlusSerializer().with_msgpack_allowlist([...])` looks like it restricts what's allowed to deserialize, but its own source early-returns `self` unchanged whenever the base serializer's `_allowed_msgpack_modules` is already `True` or `False` ŌĆö and `JsonPlusSerializer()`'s default is `True` ("allow everything, just warn"). So the original fix had never actually restricted anything, in either its first landing or since. It went unnoticed because the checkpointer was never actually reused across calls (see above) ŌĆö no checkpoint was ever really deserialized, so neither the no-op allowlist nor its gaps had ever been exercised. The earlier "verified live: zero warnings across two turns" claim was a false negative, not a real fix.

Corrected by passing `allowed_msgpack_modules` directly at `JsonPlusSerializer(...)` construction, which does work. Since the allowlist is now genuinely restrictive rather than permissive, it also needed to be complete: auditing every custom type reachable from `GraphState` surfaced two more that weren't registered ŌĆö `SearchResult` (`packages/knowledge/schemas.py`) and `Citation` (`packages/rag/schemas.py`) from the RAG fields, and `MemoryFact`/`MemoryType` (`packages/memory/schemas.py`) from `state["memories"]` ŌĆö both of which started **hard-blocking** deserialization (not just warning) the moment the allowlist became real.

**Verified live:** a real 3-turn same-conversation test ŌĆö told "remember the number 42," asked to recall it, asked to do arithmetic on it via the calculator tool ŌĆö zero warnings, zero blocked deserializations, correct recall and correct math throughout. Two real design constraints discovered and worked around, not assumed away:

1. **`AsyncPostgresSaver.from_conn_string()` is an async context manager, not a plain constructor** ŌĆö and `dependency_injector`'s async `providers.Resource` (the natural-looking DI fit) always returns an awaitable on every call, even after initialization, confirmed via a standalone test before committing to a design. Solved by using `AsyncPostgresSaver`'s plain, synchronous `__init__(conn, serde=...)` instead ŌĆö opening a real `psycopg.AsyncConnection` explicitly at app startup and overriding the DI container's `checkpointer` provider with the fully-constructed instance via `container.graph.checkpointer.override(providers.Object(checkpointer))`, in `packages/api/lifespan.py`. This is the exact same override pattern this codebase already established for per-request database sessions (`request_scoped_session`), just applied at application lifetime instead of per-request. The `MemorySaver` default stays declared as the fallback for any code path that constructs `ApplicationContainer()` directly without running through `lifespan()` (this session's own many ad-hoc verification scripts among them).
2. **A real platform bug: psycopg's async driver can't run under Windows' default `ProactorEventLoop`, and `uvicorn` hardcodes exactly that loop on native Windows regardless of any event-loop policy set at the application level** ŌĆö confirmed by reading `uvicorn.loops.asyncio.asyncio_loop_factory`'s actual source, not assumed from the error message alone. ## Chat felt slow ŌĆö measured it, found three separate real bugs, fixed all three

Reported symptom: both `POST /api/v1/chat` and `cli.py` took too long to reply. Measured directly instead of guessing: a plain "thanks, that's helpful!" message (no retrieval needed) took **7.9s**; a retrieval-routed question took **12.0s**.

**Bug 1 ŌĆö memory extraction was blocking the response for no reason.** `packages/graph/builder.py`'s graph had an `extract_memory` node between `llm` and `END`, and `ChatService._execute_runtime()`'s `await self._graph.invoke(state)` waited for the whole graph ŌĆö including that node's two LLM calls (fact extraction, then conversation summarization) ŌĆö before the HTTP response was ever built. Neither call has any effect on the reply the user is waiting for. Removed `extract_memory` from the graph entirely (`packages/graph/router.py`'s `after_llm()` now routes straight to `END`); it's still a real class (`packages/graph/nodes/extract_memory.py`, unchanged), just no longer wired into `GraphBuilder`/`GraphNodes` ŌĆö instead it's constructed via `container.graph.extract_memory()` and run as a background task, scheduled by `packages/api/routers/chat.py` (`_extract_memory_in_background()`) *after* the response is sent, for both the streaming and non-streaming paths. Opens its own fresh request-scoped session and rebuilds the message history via `ConversationContextBuilder` ŌĆö the same "never block the request path, re-fetch what you need in a fresh session" pattern already established for document ingestion (`packages/api/routers/documents.py`'s `_ingest_in_background`). **Verified: 7.9s ŌåÆ 3.2s** for the no-retrieval case, same reply content, same citations behavior.

**Bug 2 ŌĆö moving extraction to the background surfaced a real race condition.** The old synchronous graph accidentally serialized every conversation's memory summarization (one turn's full pipeline, extraction included, always finished before the next request could even start). Once extraction ran independently in the background, two turns close together in the same conversation could both call `MemoryManager.summarize()`, both check "does a summary row exist yet?", both see no, and both try to create one ŌĆö permanently corrupting the one-summary-per-conversation invariant and crashing every later summarization for that conversation with `sqlalchemy.exc.MultipleResultsFound`. Caught by deliberately firing back-to-back messages at the same conversation, not by luck. Fixed with a per-conversation `asyncio.Lock` (`packages/memory/manager.py`'s module-level `_summary_locks`, keyed by `conversation_id` since `MemoryManager` itself is a per-call `providers.Factory` and an instance-level lock wouldn't serialize anything). Also made `MemoryRepository.get_by_conversation_and_type()` (`packages/infrastructure/repositories/memory.py`) defensive regardless ŌĆö takes the most recently updated row instead of crashing if duplicates ever exist for any reason. No data cleanup was needed; the one duplicate this bug produced during testing turned out to have rolled back on its own (the losing side of the race raised inside its own transaction).

**Bug 3 ŌĆö `planner` and `load_memory` were running sequentially for no real reason, and fixing that surfaced a second, unrelated pre-existing bug.** `load_memory` only ever reads the raw last message ŌĆö it never depends on `planner`'s rewritten query ŌĆö so their strict ordering in the graph was an artifact of the order the two `add_node` calls were written in, not a real dependency. Restructured `packages/graph/builder.py` so both fan out directly from `START` and converge at a new trivial `join` node (LangGraph only runs a node once every incoming edge has fired, which is what makes `join` a safe synchronization point for `router.route()` to read `state["execution_plan"]` from). This immediately surfaced a real bug: `LoadMemoryNode` mutated the *entire* `state` dict in place and returned it, instead of returning a partial update like every other node ŌĆö harmless under the old strictly-sequential graph (only one node ever ran per step, so there was never a competing write to reconcile), but LangGraph raised `InvalidUpdateError: At key 'rewritten_query': Can receive only one value per step` the instant it ran concurrently with `planner`, since it was returning (and thus "writing") every key in `state`, including one only `planner` ever actually sets. Fixed by making `LoadMemoryNode` return `{"memories": [...]}` instead. **Verified genuinely concurrent, not just "still works"**: patched both nodes with timing instrumentation and confirmed `load_memory` and `planner` start within 2ms of each other on a real graph run.

**Also fixed, smaller but real, same session:** `POST /api/v1/chat` used to `404` when given a `conversation_id` that didn't exist yet ŌĆö by request, it now creates a new conversation under that exact client-supplied ID instead of erroring, a second, independent creation path alongside the existing omit-the-ID-for-auto-create one (`ensure_default_conversation`). And `cli.py` got several real fixes in the same pass: it was hardcoding `"stream": false` unconditionally (the server's SSE path was never actually reachable from it at all) ŌĆö now streams by default; it never sent `X-Tenant-ID`/`X-User-ID` as validated UUIDs, so a stray keystroke meant for the chat prompt could land in the Tenant/User ID prompt and silently propagate as a literal string, surfacing as a confusing `400` several steps later ŌĆö now re-prompts immediately with a clear message if the input isn't a real UUID.

Full import sweep after all three fixes: same 10 pre-existing failures, zero new ones. **One thing found but deliberately not chased in this pass**: stress-testing with three genuinely *simultaneous* requests to the same conversation (not just fast-but-sequential ones) surfaced a separate `sqlalchemy.exc.InvalidRequestError: Session is already flushing` ŌĆö unrelated to any of the three bugs above, about the request-scoped DB session itself under true concurrent requests to one conversation. Flagged, not fixed ŌĆö a different failure mode than what was reported, and out of scope for "why is chat slow."
